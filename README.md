### Three ML Components

| # | Component | Type | Purpose |
|---|-----------|------|---------|
| 1 | **Road-Closure-Risk Classifier** | LightGBM (binary) | Does this event require road closure? |
| 2 | **Congestion Clearance-Time Regressor** | Random Forest (log-target) | How long will it take to clear? |
| 3 | **Resource Recommendation Engine** | Rule-based (transparent) | Manpower, barricades, diversion plan |

## The Honest Metrics (and why they matter more than 99%)

| Model | Metric | Value |
|---|---|---|
| Road-Closure-Risk (LightGBM) | Accuracy | **~90–91%** |
| | ROC-AUC | **~0.80** |
| | 5-fold CV | **~90%**, low variance |
| | Positive class rate | **~7.4%** (realistic imbalance) |
| | Decision threshold | **0.65** (tuned) |
| Clearance-Time (Random Forest) | R² | **~0.20** |
| | MAE | **~70 minutes** |

### Why we didn't report 99.8%

We discovered a critical data leakage during our audit:
- The `priority` field is an **operational labeling rule** (not a learned outcome). Every event on a named arterial corridor is auto-labeled `High` by policy.
- `end_address` / `endlatitude` / `endlongitude` are only populated **after** a closure is logged.

A naive model on these fields scores **99.8% accuracy by reading back a database rule**, not by forecasting anything. We permanently excluded them and rebuilt with only information genuinely available at event-report time. **A defensible ~90% on a genuinely hard task is stronger evidence of real data science than an inflated number on a circular one.**

## Quick Start

### Option A: Docker (Recommended)

```bash
# 1. Place your Astram CSV in the project root
cp /path/to/Astram_event_data_anonymized.csv ./

# 2. Start everything
docker-compose up

# Frontend: http://localhost:3000
# Backend API: http://localhost:8000
# API Docs: http://localhost:8000/docs
```

### Option B: Manual

**Backend:**
```bash
cd backend
pip install -r requirements.txt
python train.py                          # trains models, saves to models/
uvicorn main:app --reload --port 8000
```

**Frontend:**
```bash
cd frontend
npm install
npm run dev   # http://localhost:3000
```

## Feature Engineering (Single Source of Truth)

All 26 features for the closure classifier are computed **server-side** in `features.py`. The frontend only sends raw inputs.

**Temporal (11):** `hour`, `dow`, `month`, `is_weekend`, `is_morning_peak` (8–11AM), `is_evening_peak` (5–8PM), `is_night`, `hour_sin`, `hour_cos`, `dow_sin`, `dow_cos`

**Spatial (1):** `geo_cluster` (KMeans k=25 on lat/lon)

**Context (4):** `authenticated_flag`, `event_type_flag`, `has_junction`, `description_len`

**Target-encoded (leakage-safe, 6):** `corridor_te`, `police_station_te`, `event_cause_te`, `geo_cluster_te`, `zone_te`, `veh_type_te`

**Label-encoded (4):** `event_cause_enc`, `veh_type_enc`, `corridor_enc`, `zone_enc`

**Permanently excluded** (per leakage audit): `priority`, `end_address`, `endlatitude`, `endlongitude`, `corridor_event_freq`, `ps_event_freq`, any duration-derived field.
